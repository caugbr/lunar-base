Introdução

A pasta /config do Lunar Base contém os arquivos que definem o comportamento central do sistema. Diferente dos plugins (que são modulares e podem ser ativados/desativados), estes arquivos são parte do core e devem ser editados com cuidado.

Quando editar cada arquivo

Arquivo Frequência de Edição Risco
admin.php Baixa (apenas na customização inicial) Baixo
defaultUsers.php Nunca (gerado automaticamente) -
imageSizes.php Média (conforme necessidade de layouts) Baixo
pageTemplates.php Baixa (ao criar novos templates) Baixo
postTemplates.php Baixa (ao criar novos templates) Baixo
rolesPermissions.php Muito baixa (apenas para criar novos perfis) Alto
settings.php Média (ao adicionar novas configurações) Médio
site.php Baixa (se não usar o plugin Menus) Baixo
Importante: Sempre faça backup dos arquivos antes de editar. Um erro de sintaxe PHP pode deixar o sistema inacessível.

admin.php (Menu e Skin da Admin)

Define a estrutura do menu lateral da área administrativa e o tema visual (skin) usado no painel.

Estrutura do Menu

O menu é organizado em seções, cada uma contendo itens de navegação:

Exemplo de item do menu

[
    'label'      => 'Páginas',
    'icon'       => 'file',
    'route'      => 'admin.pages.index',
    'active'     => 'admin.pages.*',
    'permission' => 'manage-pages',
]

Parâmetros de cada item

Parâmetro Obrigatório Descrição
label Sim Texto exibido no menu
icon Sim Nome do ícone Lucide (ex: file, users)
route Sim Rota Laravel ao clicar
active Sim Padrão que marca o item como ativo (use * como wildcard)
permission Não Exibe apenas se o usuário tiver essa permissão
role Não Exibe apenas se o usuário tiver esse perfil (ex: admin)

Dashboard

Array que define a estrutura da apresentação do dashboard

Configuração padrão

"dashboard" =>  [
    "title" => "Dashboard",
    "subtitle" => "Visão geral do sistema",
    "icon" => "layout-dashboard",
    "cardTitle" => "Visão geral",
    "columns" => 4,
    "boxView" => "admin.dashboard.box"
]

Parâmetros

Parâmetro Descrição
title Título no header da página
subtitle Texto abaixo do título no header
icon Ícone no título do card
cardTitle Título dentro do card
columns Quantidade de colunas para os dashboard widgets
boxView View a ser usada para renderizar os widgets

Ícones Disponíveis

Use nomes do Lucide Icons. Exemplos comuns:

layout-dashboard, file, files, image, tags, form,
users, user-pen, user-key, settings, list-checks, puzzle, palette

Skin da Admin

A chave skin define qual arquivo CSS será carregado de public/css/admin/skins/:

Exemplo

"skin" => "default"

Se o valor for "default", o sistema carrega css/admin/vars.css. Qualquer outro valor (ex: "dark") carrega css/admin/skins/vars-dark.css.

Dica: Para criar um novo skin, duplique o arquivo public/css/admin/vars.css e salve em public/css/admin/skins/, com o nome vars-skin-name.css personalize as variáveis CSS e adicione o nome no admin.php. O nome deve ser a string entre 'vars-' e '.css' ('skin-name').

defaultUsers.php (Usuários Padrão)

Este arquivo não deve ser editado manualmente. Ele trabalha em conjunto com o script install.sh para gerar os usuários iniciais do sistema durante a instalação.

Como funciona

  1. O script install.sh lê o rolesPermissions.php para identificar os perfis existentes.
  2. Gera um usuário para cada perfil (admin, editor, author, subscriber).
  3. Salva os dados em storage/app/.install/default_users_data.json.
  4. O seeder usa esse JSON para popular o banco na instalação.

Senhas padrão

Todos os usuários gerados têm a senha: Pass#1029

Recomendação: Após a instalação, altere as senhas imediatamente pelo painel administrativo.

imageSizes.php (Tamanhos de Imagem)

Define tamanhos extras de imagem que serão gerados automaticamente ao fazer upload. O sistema já cria dois tamanhos padrão independentemente deste arquivo:

  • Thumbnail: definido nas configurações do sistema (Admin → Configurações → Mídia)
  • Large: 1200px de largura, usado para SEO e compartilhamento social

Estrutura de definição

Exemplo de tamanho personalizado

"medium" => [
    'width' => 600,
    'height' => null,      // null = proporcional
    'crop' => false,       // false = redimensiona mantendo proporção
    'position' => null,    // só usado se crop = true
]

Parâmetros disponíveis

Parâmetro Tipo Descrição
width int Largura em pixels (obrigatório)
height int|null Altura em pixels (null = proporcional)
crop bool Se true, corta a imagem; se false, redimensiona
position string Posição do corte: center, top, bottom (só se crop = true)

Exemplos práticos

Tamanhos comuns

return [
    // Banner largo (proporcional)
    "banner" => [
        'width' => 1920,
        'height' => null,
        'crop' => false,
    ],

    // Card quadrado (corte centralizado)
    "card-square" => [
        'width' => 400,
        'height' => 400,
        'crop' => true,
        'position' => 'center',
    ],

    // Thumbnail vertical (corte no topo)
    "portrait" => [
        'width' => 300,
        'height' => 450,
        'crop' => true,
        'position' => 'top',
    ],
];
Como usar: Após definir um tamanho, ele estará disponível via $media->getUrl('card-square') nas views.

pageTemplates.php (Templates de Páginas)

Define quais templates estarão disponíveis no dropdown ao criar/editar uma página pública.

Estrutura

Formato do array

return [
    'templates' => [
        'default' => 'Site page',
        'page' => 'Default',
        'fullwidth' => 'Fullwidth',
        'left-sidebar' => 'Left Sidebar'
    ],
    'default' => 'default',
];

Parâmetros

Chave Descrição
templates Array de nome_do_arquivo => Rótulo visível
default Template padrão (usado se nenhum for selecionado)

Local dos arquivos de template

Os templates devem estar em resources/views/public/page-templates/:

resources/views/public/page-templates/
    ├── default.blade.php
    ├── page.blade.php
    ├── fullwidth.blade.php
    └── left-sidebar.blade.php

Criando um novo template

1Crie o arquivo Blade

Em resources/views/public/page-templates/meu-template.blade.php

2Registre no config

'templates' => [
    'default' => 'Site page',
    'meu-template' => 'Meu Template Personalizado',
],

3Use no template

O conteúdo da página estará disponível na variável $page:

@extends('public.site-layout')

@section('content')
<article class="page">
    <h1>{{ $page->title }}</h1>
    <div class="page-content">
        {!! $page->content !!}
    </div>
</article>
@endsection

postTemplates.php (Templates de Posts)

Funciona exatamente como o pageTemplates.php, mas para posts do blog.

Estrutura padrão

return [
    'templates' => [
        'default' => 'Post padrão',
    ],
    'default' => 'default',
];

Local dos arquivos

resources/views/public/post-templates/
└── default.blade.php

Exemplo de template de post

@extends('public.site-layout')

@section('content')
<article class="post">
    <header>
        <h1>{{ $post->title }}</h1>
        <time>{{ $post->published_at->format('d/m/Y') }}</time>
    </header>

    <div class="post-content">
        {!! $post->content !!}
    </div>

    @if($post->tags->count())
    <div class="post-tags">
        @foreach($post->tags as $tag)
        <span class="tag">{{ $tag->name }}</span>
        @endforeach
    </div>
    @endif
</article>
@endsection
Variáveis disponíveis: $post (model completo), $comments (se plugin ativo), hooks automáticos como post.after_content.

rolesPermissions.php (Permissões e Perfis)

Este é o arquivo mais crítico do sistema. Define quem pode fazer o quê através de perfis (roles) e permissões (permissions). É a fonte da verdade do controle de acesso.

Cuidado: Erros neste arquivo podem travar o acesso administrativo ou expor funcionalidades sensíveis. Teste sempre em ambiente de desenvolvimento primeiro.

Estrutura do arquivo

O arquivo tem três seções principais:

Visão geral

return [
    'roles' => [...],              // 1. Define os perfis
    'permissionsByRole' => [...],  // 2. Mapeia permissões por perfil
    'permissionGroups' => [...],   // 3. Agrupa permissões para a UI
];

1. Roles (Perfis)

Cada perfil tem um slug (chave única), name (exibido) e description:

Perfis padrão

'roles' => [
    'admin' => [
        'name' => 'Administrador',
        'description' => 'Acesso total ao sistema.',
    ],
    'editor' => [
        'name' => 'Editor',
        'description' => 'Gerencia publicações.',
    ],
    'author' => [
        'name' => 'Autor',
        'description' => 'Gerencia próprias publicações.',
    ],
    'subscriber' => [
        'name' => 'Assinante',
        'description' => 'Acesso mínimo.',
    ],
],

2. Permissions by Role

Define quais permissões cada perfil possui. O admin deve ter todas as permissões:

Exemplo

'permissionsByRole' => [
    'admin' => [
        'view-dashboard',
        'manage-users',
        'manage-pages',
        'manage-posts',
        'manage-settings',
        // ... todas as permissões
    ],
    'editor' => [
        'view-dashboard',
        'manage-pages',
        'manage-posts',
    ],
    'author' => [
        'view-dashboard',
        'manage-own-pages',
        'manage-own-posts',
    ],
],
Regra crítica: O perfil admin DEVE conter todas as permissões listadas em permissionGroups. Caso contrário, o administrador perderá acesso a funcionalidades.

3. Permission Groups

Agrupa permissões por área para exibição na interface de gerenciamento:

Exemplo

'permissionGroups' => [
    'dashboard' => [
        'view-dashboard' => 'Ver painel de controle',
    ],
    'users' => [
        'manage-users' => 'Gerenciar usuários',
    ],
    'publications' => [
        'manage-pages' => 'Criar e editar todas as páginas',
        'manage-posts' => 'Criar e editar todos os posts',
        'manage-own-pages' => 'Criar e editar suas próprias páginas',
        'manage-own-posts' => 'Criar e editar seus próprios posts',
    ],
],

Permissões disponíveis

Permissão Descrição
view-dashboard Acessar o painel principal
manage-users Criar, editar, excluir qualquer usuário
view-reports Ver relatórios e estatísticas
edit-profile Alterar dados do próprio perfil
manage-settings Modificar configurações globais
manage-pages Criar e editar páginas de qualquer autor
manage-posts Criar e editar posts de qualquer autor
manage-own-pages Criar e editar apenas páginas próprias
manage-own-posts Criar e editar apenas posts próprios

Criando um novo perfil

1Adicione o role

'roles' => [
    // ... perfis existentes
    'moderator' => [
        'name' => 'Moderador',
        'description' => 'Aprova comentários e gerencia conteúdo.',
    ],
],

2Defina as permissões

'permissionsByRole' => [
    // ... outros perfis
    'moderator' => [
        'view-dashboard',
        'edit-profile',
        'view-reports',
    ],
],

3Execute o seeder

php artisan db:seed --class=RolePermissionSeeder

Usando no código

Verificando permissão

// No controller
if (auth()->user()->permission('manage-pages')) {
    // permite ação...
}

// Na view Blade
@if(auth()->user()->permission('manage-users'))
    <a href="/admin/users">Usuários</a>
@endif

Protegendo rotas

// Por permissão
Route::middleware('permission:manage-users')->group(function () {
    // rotas protegidas...
});

// Por role
Route::middleware('role:admin')->group(function () {
    // rotas protegidas...
});

settings.php (Configurações do Sistema)

Define as configurações administráveis que aparecem em Admin → Configurações. Cada grupo vira uma seção ou aba na interface.

Estrutura de um grupo

Formato básico

'general' => [
    'tab' => 'Geral',                    // Opcional: nome da aba
    'title' => 'Configurações Gerais',   // Obrigatório
    'description' => 'Descrição...',     // Opcional
    'icon' => 'settings',                // Opcional: ícone Lucide
    'fields' => [
        // ... campos aqui
    ],
],

Tipos de campo disponíveis

Tipo Uso Exemplo
subtitle Título de subseção (H3) Organização visual
paragraph Texto explicativo Instruções ao usuário
text Campo de texto simples Nome do site
textarea Texto longo Descrição do site
number Número com min/max Itens por página
email E-mail validado E-mail de contato
url URL validada Link do Facebook
password Senha (mascarada) Senha SMTP
image Upload de imagem Logo do site
switch Ligar/desligar Ativar CAPTCHA
select Dropdown Tema padrão
radio Opções exclusivas Formato de imagem
checkbox Múltipla escolha Categorias
icon Seletor de ícone Lucide Ícone de menu
page Seletor de página Página de privacidade

Configurações avançadas por campo

depends_on (condicional)

[
    'key' => 'turnstile_site_key',
    'type' => 'text',
    'label' => 'Turnstile Site Key',
    'depends_on' => [
        'field' => 'use_captcha',
        'operator' => '===',
        'value' => true,
    ],
]

O campo só aparece se use_captcha estiver ativado.

warn_on_change (aviso)

[
    'key' => 'pages_base',
    'type' => 'text',
    'label' => 'Namespace para páginas',
    'warn_on_change' => 'Ao mudar este valor, as URLs de todas as páginas irão mudar.',
]

Exibe modal de confirmação ao alterar o valor.

Usando as settings no código

No PHP

$siteName = setting('general.site_name', 'Lunar Base');
$useCaptcha = setting('auth.use_captcha', false);

No Blade

<h1>{{ setting('general.site_name') }}</h1>

@if(setting('auth.use_captcha'))
    @include('partials.captcha')
@endif

Grupos padrão do sistema

Grupo Descrição
general Nome, descrição, tema, obrigações legais
navigation Breadcrumbs, namespaces de URL
auth CAPTCHA, 2FA
reading Tamanho de resumo, paginação, acessibilidade
media Thumbnails, qualidade, compressão
social Links de redes sociais
mail Configurações SMTP
Dica: Plugins também podem injetar configurações dinamicamente via Settings::add() no Service Provider.

site.php (Configurações do Frontend)

Define configurações específicas do site público. No momento, contém apenas a definição do menu principal padrão.

Nota: Você pode usar o plugin Menus para criar um menu para substituir as definições, neste case o array nesse arquivo é ignorado. O plugin Menus permite criar menus dinâmicos via painel administrativo.

Estrutura do menu

Formato padrão

return [
    "mainMenu" => [
        [
            "label" => "Início",
            "route" => 'home'
        ],
        [
            "label" => "Blog",
            "path" => "blog"
        ],
        [
            "label" => "Sobre",
            "slug" => "sobre-nos"
        ],
    ]
];

Três formas de definir o destino

1. Rota Laravel

[
    "label" => "Início",
    "route" => "home"  // Rota nomeada do Laravel
]

Use para páginas estáticas do sistema (home, blog.index, etc.)

2. Slug (página ou post dinâmico)

[
    "label" => "Sobre Nós",
    "slug" => "sobre-nos"
]

Para páginas/posts criados na administração. O sistema resolve a URL automaticamente.

3. URL livre

[
    "label" => "Google",
    "path" => "https://google.com"
]

Para links externos ou caminhos customizados.

Parâmetro namespace (opcional)

Para páginas, você pode usar namespace para desambiguizar páginas com o mesmo slug:

[
    "label" => "Contato",
    "slug" => "contato",
    "namespace" => "institucional"  // Só para páginas
]

Regras importantes

  • Cada item deve ter apenas uma forma de destino (route, slug ou path)
  • namespace só funciona com slug e apenas para páginas (não posts)
  • A ordem no array define a ordem de exibição no menu

Quando usar este arquivo vs Plugin Menus

Cenário Solução Recomendada
Menu simples e estático Editar site.php
Menu que muda frequentemente Plugin Menus
Sub-menus e dropdowns Plugin Menus
Menus diferentes por página Plugin Menus
Links dinâmicos (anti-404) Plugin Menus

Boas Práticas

1. Versionamento

Mantenha os arquivos de config no versionamento (Git), mas nunca commit dados sensíveis como senhas SMTP. Use variáveis de ambiente (.env) para isso.

2. Backup antes de editar

# Faça backup antes de alterar
cp config/rolesPermissions.php config/rolesPermissions.php.backup

# Em caso de erro, restaure
cp config/rolesPermissions.php.backup config/rolesPermissions.php

3. Teste em desenvolvimento

Sempre teste alterações em ambiente local antes de aplicar em produção. Um erro de sintaxe pode deixar o site inacessível.

4. Documente mudanças

Adicione comentários nos arquivos explicando o motivo de customizações:

// Adicionado em 2026-01-15 por João
// Novo perfil para moderadores de comentários
'moderator' => [
    'name' => 'Moderador',
    'description' => 'Aprova comentários.',
],

5. Use o sistema de permissões

Não hardcode verificações de role no código. Use as permissões definidas no rolesPermissions.php:

✅ Correto

@if(auth()->user()->permission('manage-pages'))
    <a href="/admin/pages">Páginas</a>
@endif

❌ Evite

@if(auth()->user()->role === 'admin')
    <a href="/admin/pages">Páginas</a>
@endif

6. Mantenha o admin.php sincronizado

Se criar uma nova permissão, atualize também o menu admin para que os usuários vejam o link:

// Em rolesPermissions.php
'permissionGroups' => [
    'reports' => [
        'view-analytics' => 'Ver analytics',
    ],
],

// Em admin.php
'items' => [
    [
        'label' => 'Analytics',
        'icon' => 'bar-chart',
        'route' => 'admin.analytics',
        'active' => 'admin.analytics.*',
        'permission' => 'view-analytics',  // Mesma permissão!
    ],
]

7. Cache de configurações

Em produção, use cache de config para performance:

# Gerar cache (deploy)
php artisan config:cache

# Limpar cache (desenvolvimento)
php artisan config:clear
Atenção: Com cache ativo, mudanças nos arquivos de config só terão efeito após limpar o cache.
