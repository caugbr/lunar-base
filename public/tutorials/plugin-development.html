Introdução

O sistema de plugins do Lunar Base permite estender as funcionalidades do sistema de forma modular. Cada plugin funciona como uma aplicação isolada, possuindo suas próprias rotas, controladores, visões, modelos e migrações.

Comando de Geração Rápida

O Lunar Base oferece um comando Artisan que gera a estrutura completa de um plugin automaticamente, incluindo diretórios, arquivos base, manifesto e link simbólico para assets públicos.

Uso Básico:

BASHphp artisan plugin:create "NomeDoPlugin" "Descrição opcional do plugin"

Opções Interativas:

Durante a execução, o comando fará três perguntas interativas:

  • Requer banco de dados? - Cria a pasta Models/ e database/migrations/, além de gerar um modelo e uma migration em branco.
  • Requer Controller e Rotas? - Cria a pasta Http/Controllers/, gera um controller base e o arquivo routes.php com middleware web e auth.
  • Requer Views? - Cria a pasta resources/views/ e resources/help-views/, gerando uma view de exemplo e uma view de ajuda para o painel administrativo.

O que é gerado automaticamente:

plugins/
└── NomeDoPlugin/
    ├── Http/Controllers/NomeDoPluginController.php  (se confirmado)
    ├── Models/NomeDoPlugin.php                      (se confirmado)
    ├── database/migrations/2026_..._create_nome_do_plugin_table.php  (se confirmado)
    ├── resources/views/index.blade.php              (se confirmado)
    ├── resources/help-views/help.blade.php          (sempre)
    ├── resources/assets/css/                      (sempre)
    ├── resources/assets/js/                       (sempre)
    ├── plugin.json
    ├── NomeDoPluginServiceProvider.php
    └── routes.php                                   (se confirmado)
Dica: O comando também cria automaticamente um link simbólico (ou junction no Windows) em public/plugins/nome-do-plugin apontando para plugins/NomeDoPlugin/resources/assets, permitindo que você referencie assets públicos sem configuração manual.

Estrutura de Pastas

Os plugins residem na pasta /plugins na raiz do projeto. A estrutura padrão de um plugin chamado "MeuPlugin" é:

plugins/
└── MeuPlugin/
    ├── Http/
    │   └── Controllers/
    │       └── MeuController.php
    ├── Models/
    │   └── MeuModel.php
    ├── database/
    │   └── migrations/
    │       └── 2026_01_01_000000_create_meu_plugin_table.php
    ├── resources/
    │   ├── views/
    │   │   ├── admin/
    │   │   └── public/
    │   └── assets/
    │       ├── css/
    │       └── js/
    ├── plugin.json
    ├── MeuPluginServiceProvider.php
    └── routes.php

O Manifesto (plugin.json)

O arquivo plugin.json armazena os metadados necessários para que o painel do administrador reconheça o plugin:

JSON{
    "name": "MeuPlugin",
    "description": "Adiciona funcionalidades personalizadas ao sistema.",
    "version": "1.0.0",
    "provider": "Plugins\MeuPlugin\MeuPluginServiceProvider"
}

Ciclo de Vida (ServiceProvider)

A classe ServiceProvider é a porta de entrada do plugin no ciclo de inicialização do Laravel. Ela determina o que é registrado (bindings) e o que é carregado (assets, views, rotas) em cada requisição.

Boas Práticas de Organização

  • Nomenclatura: Use sempre PascalCase para pastas e namespaces (ex: Plugins\MeuPlugin).
  • Namespaces de Views: Sempre carregue suas views com um identificador de namespace exclusivo (como meuplugin) no seu ServiceProvider para evitar colisões com outros módulos.
  • Sanitização: Ao processar dados enviados por usuários no frontend, utilize funções de higienização do PHP (como strip_tags) para evitar falhas de segurança.
