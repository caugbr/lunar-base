O que são Rotas Dinâmicas

Rotas dinâmicas permitem que seu plugin exponha URLs amigáveis no site público sem modificar arquivos de roteamento do Laravel. Elas são resolvidas em tempo de execução pelo RouteOrchestratorController, que atua como um roteador central do Lunar Base.

Ideal para: galerias, lojas, formulários públicos, agendas, catálogos, ou qualquer conteúdo gerenciado pelo plugin que precise de uma URL acessível diretamente pelo visitante.

Limitações de URL

⚠️ Atualmente apenas 1 a 3 segmentos são suportados nativamente.

O orquestrador captura URLs com 1, 2 ou 3 segmentos via rotas declaradas no web.php. Seu plugin não pode registrar rotas como:

  • /minha-rota/com/quatro/segmentos ❌
  • /a/b/c/d/e ❌
🔧 Estendendo para N segmentos:

É possível remover essa limitação adicionando uma rota catch-all no final do routes/web.php:

PHP// routes/web.php - adicionar no final, após todas as outras rotas
Route::get('/{any}', [RouteOrchestratorController::class, 'handleCatchAll'])
    ->where('any', '.*')
    ->name('dynamic.catch.all');

E implementando o método no orquestrador:

PHPpublic function handleCatchAll(Request $request, $any)
{
    $url = trim($any, '/');

    if ($view = DynamicRoutes::resolve($url)) {
        return $view;
    }

    abort(404);
}

Com essa adição, plugins podem registrar rotas com qualquer número de segmentos, desde que não conflitem com rotas estáticas do Laravel ou com as regras do CMS (que têm prioridade por estarem declaradas antes).

Como Registrar

Utilize o método estático DynamicRoutes::register() dentro do boot() do Service Provider do seu plugin:

PHPuse App\Support\DynamicRoutes;

// No boot() do seu ServiceProvider:
DynamicRoutes::register('GET', 'galeria/{slug}', function ($slug) {
    $item = \Plugins\MinhaGaleria\Models\Item::where('slug', $slug)->firstOrFail();

    return view('minha-galeria::public.show', compact('item'));
});
Assinatura do método: DynamicRoutes::register(string $method, string $path, callable $callback)
  • $method - Método HTTP: GET, POST, PUT, DELETE
  • $path - Caminho relativo, sem barra inicial. Use {nome} para parâmetros.
  • $callback - Função ou callable que receberá os parâmetros capturados na ordem em que aparecem na URL.

Como o Core Resolve as URLs

O Laravel possui rotas estáticas declaradas no routes/web.php (como /, /docs, /admin/*). Se a URL não corresponder a nenhuma delas, o orquestrador entra em ação através de rotas catch-all que capturam 1, 2 ou 3 segmentos:

PHP (routes/web.php)Route::get('/{base}/{namespace}/{slug}', [...'handleThreeSegments']);
Route::get('/{base}/{slug}', [...'handleTwoSegments']);
Route::get('/{base}', [...'handleOneSegment']);

Dentro do orquestrador, a decisão segue esta ordem exata:

  1. 1. Rotas do CMS (Blog e Páginas)

    O orquestrador verifica se a URL corresponde a:

    • Base do blog (/blog por padrão, mas consulte setting('navigation.blog_base')) → listagem de posts
    • Post individual (/blog/{slug}) → post do blog
    • Página com base fixa (/page/{slug} por padrão, mas consulte setting('navigation.pages_base')) → página do CMS
    • Página com namespace (sem base) → /{namespace}/{slug} (somente se pages_base for null)
    • Página com base + namespace → /page/{namespace}/{slug}
    • Taxonomia do blog → /blog/{taxonomia}/{termo}
  2. 2. Rotas Dinâmicas de Plugins

    Se nenhuma regra do CMS corresponder, o orquestrador percorre todas as rotas registradas via DynamicRoutes::register() na ordem em que foram adicionadas. Para cada rota:

    • Compara o método HTTP da requisição com o método registrado
    • Aplica uma regex gerada a partir do padrão (ex: galeria/{slug} vira #^galeria/([^/]+)$#)
    • Se houver match, extrai os grupos capturados e executa o callback
  3. 3. 404 - Nada Encontrado

    Se nenhuma rota dinâmica corresponder, o orquestrador dispara abort(404).

⚠️ Atenção: Suas rotas de plugin são verificadas depois de todas as regras do CMS. Isso significa que se um usuário criar uma página com slug que conflite com seu prefixo, sua rota será interceptada pelo CMS e nunca chegará ao plugin.

Parâmetros de URL

Use a sintaxe {nome} no path para definir segmentos variáveis. Cada parâmetro se torna um argumento do callback, na mesma ordem:

PHP// URL: /eventos/2024/festival-de-inverno
// Path:  eventos/{ano}/{slug}

DynamicRoutes::register('GET', 'eventos/{ano}/{slug}', function ($ano, $slug) {
    // $ano  = '2024'
    // $slug = 'festival-de-inverno'

    $evento = Evento::where('ano', $ano)->where('slug', $slug)->firstOrFail();

    return view('eventos::public.detalhe', compact('evento'));
});
Limitações dos parâmetros:
  • Cada {param} captura apenas um segmento (não permite barras)
  • Use nomes descritivos - eles não afetam o funcionamento, apenas a legibilidade
  • Não há validação automática de tipo; faça no callback (ex: is_numeric($ano))

Precedência e Conflitos

Ordem de Resolução (do mais prioritário ao menos)

  1. Rotas estáticas do Laravel - /, /docs, /admin/*, etc.
  2. Blog base - valor de setting('navigation.blog_base')
  3. Posts do blog - /{blog_base}/{slug}
  4. Páginas com base fixa - /{pages_base}/{slug}
  5. Páginas com namespace (sem base) - /{namespace}/{slug} (somente se pages_base for null)
  6. Páginas com base + namespace - /{pages_base}/{namespace}/{slug}
  7. Taxonomias do blog - /{blog_base}/{taxonomia}/{termo}
  8. Rotas dinâmicas de plugins - na ordem de registro
⚠️ Slugs críticos a evitar nos seus plugins:

Antes de escolher o prefixo da sua rota, consulte as configurações atuais do site para evitar colisão:

PHP$blogBase  = setting('navigation.blog_base', 'blog');
$pagesBase = setting('navigation.pages_base', 'page');
$postsBase = setting('navigation.posts_base', 'post');

// NUNCA use esses valores como prefixo de rota do seu plugin

Esses valores são editáveis pelo administrador do site. Para maior segurança, prefira prefixos únicos e específicos do seu plugin: /galeria, /loja, /agenda, /catalogo, etc.

Conflito entre Plugins

Se dois plugins registrarem o mesmo path, o primeiro a ser registrado ganha. A ordem de registro depende da ordem de carregamento dos Service Providers, que segue a ordem definida em config/app.php ou pelo auto-discovery do Composer.

Exemplos Práticos

Exemplo 1: Rota Simples sem Parâmetros

PHPDynamicRoutes::register('GET', 'vagas-trabalho', function () {
    $vagas = \Plugins\Recrutamento\Models\Vaga::ativas()->get();

    return view('recrutamento::public.lista', compact('vagas'));
});

Acessível em: https://seusite.com/vagas-trabalho

Exemplo 2: Rota com Um Parâmetro

PHPDynamicRoutes::register('GET', 'catalogo/{slug}', function ($slug) {
    $item = \Plugins\MeuCatalogo\Models\Item::where('slug', $slug)->firstOrFail();

    return view('meu-catalogo::public.show', compact('item'));
});

Acessível em: https://seusite.com/catalogo/item-exemplo

Exemplo 3: Rota com Múltiplos Parâmetros

PHPDynamicRoutes::register('GET', 'loja/{categoria}/{produto_slug}', function ($categoria, $produtoSlug) {
    $produto = \Plugins\Loja\Models\Produto::where('slug', $produtoSlug)
        ->whereHas('categoria', fn ($q) => $q->where('slug', $categoria))
        ->firstOrFail();

    return view('loja::public.produto', compact('produto', 'categoria'));
});

Acessível em: https://seusite.com/loja/eletronicos/celular-smartphone

Exemplo 4: Rota POST (Formulário Público)

PHPDynamicRoutes::register('POST', 'contato/enviar', function () {
    $dados = request()->validate([
        'nome'  => 'required|string|max:255',
        'email' => 'required|email',
        'mensagem' => 'required|string',
    ]);

    // Processa o envio...
    Mail::to(setting('site.email'))->send(new ContatoMail($dados));

    return redirect()->back()->with('success', 'Mensagem enviada!');
});

O formulário HTML deve apontar para: https://seusite.com/contato/enviar

Exemplo 5: Tratamento de 404 no Callback

PHPDynamicRoutes::register('GET', 'galeria/{slug}', function ($slug) {
    $galeria = \Plugins\MinhaGaleria\Models\Galeria::where('slug', $slug)->first();

    if (!$galeria || !$galeria->publicada) {
        abort(404, 'Galeria não encontrada ou não publicada.');
    }

    return view('minha-galeria::public.show', compact('galeria'));
});

Boas Práticas

  • Use prefixos únicos do plugin. Evite termos genéricos como /item, /detalhe, /show. Prefira /catalogo, /galeria, /agenda, etc.
  • Consulte as bases configuradas antes de definir suas rotas. Use setting('navigation.blog_base'), setting('navigation.pages_base') e setting('navigation.posts_base') para saber quais prefixos estão reservados no ambiente atual.
  • Respeite o limite de 3 segmentos (ou implemente o catch-all). Se sua lógica precisar de mais profundidade e o catch-all ainda não estiver implementado, reestruture a URL. Use query strings ou codifique a hierarquia no slug: /produto/eletronicos-celulares-samsung-galaxy em vez de 4 segmentos.
  • Valide os parâmetros no callback. O regex só garante que há um valor; a validação semântica (ex: slug existe, está publicado) é responsabilidade do plugin.
  • Documente as rotas do seu plugin. Inclua na documentação do plugin quais URLs ele reserva, para que o administrador do site não crie páginas conflitantes no CMS.
  • Use firstOrFail() ou abort(404) explicitamente. Se o registro não for encontrado, retorne 404 para manter o comportamento esperado do Laravel.
